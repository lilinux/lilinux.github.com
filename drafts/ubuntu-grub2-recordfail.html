ubuntu崩溃重启后卡在grub界面

日期 2014-08-05 13:49:00 作者 lilinux 类别 Linux. 标签 ubuntu grub recordfail timeout

原文地址：http://blog.lilinux.com/drafts/ubuntu-grub2-recordfail.html

今天同事说，ubuntu子机重启失败，ssh和ping都不通。

我找到母机一看，qemu进程正好绑了个vnc端口，然后通过多层ssh dynamic forwarding，登录上了子机的vnc界面，发现vnc显示着grub，正在等待选择启动项。

按下回车，问题就解决了。

可是，grub都有超时时间，为什么在重启之后，grub没有在超时时间到期后自动选择默认的启动项呢？

于是我想到了我安身立命的工具之一 grep

# grep timeout /boot/grub/grub.cfg
set timeout=-1
  set timeout=10
if [ "x${timeout}" != "x-1" ]; then
      set timeout=-1
      set timeout=0
      set timeout=0

果不出我所料，set timeout=-1不就相当于禁止了自动启动了吗。

进一步分析/boot/grub/grub.cfg文件，发现有两种可能会导致set timeout=-1

  1. 当引导时按住了键盘上的Shift键

    :::bash if [ "x${timeout}" != "x-1" ]; then if keystatus; then if keystatus --shift; then set timeout=-1 else set timeout=0 fi else if sleep --interruptible 3 ; then set timeout=0 fi fi fi

  2. 当recordfail被设置

    :::bash if [ "${recordfail}" = 1 ]; then set timeout=-1 else set timeout=10 fi

然后我掐指一算，同事不可能按得往Shift的，那一定就是 recordfail了

解决的办法是把第二段代码改成

if [ "${recordfail}" = 1 ]; then
  set timeout=10
else
  set timeout=10
fi

等等，这样做会不会即难看又留下隐患呢，当升级内核时，apt-get会调用update-grub生成grub.cfg文件，那说明这个文件应该是由配置模板和数据通过update-grub生成的。

我们可以看到grub.cfg的开头就这几句话

#
# DO NOT EDIT THIS FILE
#
# It is automatically generated by grub-mkconfig using templates
# from /etc/grub.d and settings from /etc/default/grub
#

嗯，果然如此，搜索recordfail相关的网页找到一个页面

正规的修改方式是

echo 'GRUB_RECORDFAIL_TIMEOUT=10' >> /etc/default/grub
update-grub
